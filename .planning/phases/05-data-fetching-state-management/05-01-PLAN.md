  <behavior>
    - QueryClient exports a singleton with staleTime 5min, gcTime 30min, retry 1, refetchOnWindowFocus false, mutation retry 0
    - Item type has id (UUID string), user_id, title, description (nullable), category, priority, status, created_at, updated_at
    - CreateItemInput omits id, user_id, created_at, updated_at from Item
    - UpdateItemInput is Partial of CreateItemInput with required id
    - SQL creates items table with all columns, enables RLS, creates SELECT/INSERT/UPDATE/DELETE policies scoped to auth.uid() = user_id
    - SQL creates seed_user_items() trigger function (SECURITY DEFINER) on auth.users insert that inserts 30-50 demo items for the new user
    - Demo items use "tasks" as entity with categories: frontend, backend, design, devops; priorities: high, medium, low; statuses: active, completed, archived
  </behavior>
  <action>
    1. Install TanStack Query: `cd apps/mobile && npm install @tanstack/react-query --legacy-peer-deps`

    2. Create `src/lib/query-client.ts`: Export a QueryClient singleton with defaults from RESEARCH.md Pattern 1:
       - staleTime: 5 minutes, gcTime: 30 minutes, retry: 1, refetchOnWindowFocus: false
       - Mutations: retry: 0

    3. Create `src/types/database.ts`: Define the Item interface matching the Supabase table schema:
       - id: string (UUID), user_id: string, title: string, description: string | null, category: string, priority: string, status: string, created_at: string, updated_at: string
       - Export CreateItemInput (Omit<Item, 'id' | 'user_id' | 'created_at' | 'updated_at'>)
       - Export UpdateItemInput (Partial<CreateItemInput> & { id: string })
       - Export CATEGORIES, PRIORITIES, STATUSES as const arrays for filter dropdowns

    4. Create `infra/supabase/migrations/20260310120000_create_items_table.sql`:
       - Use the "tasks" entity (title, description, category, priority, status)
       - Table with UUID primary key, user_id FK to auth.users(id) ON DELETE CASCADE
       - RLS enabled with 4 policies (SELECT, INSERT, UPDATE, DELETE) all scoped to auth.uid() = user_id
       - Indexes on user_id and category columns
       - Do NOT use pg_trgm (may not be enabled); simple ILIKE will suffice for search
       - seed_user_items() trigger function with SECURITY DEFINER that inserts 35 varied demo tasks on auth.users INSERT
       - Demo tasks should have realistic titles across categories (e.g., "Implement dark mode toggle", "Set up CI/CD pipeline", "Design onboarding flow", "Configure load balancer") with varied priorities and statuses

    5. Update `app/_layout.tsx`: Import QueryClientProvider from @tanstack/react-query, import queryClient from @/lib/query-client, import ToastProvider and ModalProvider (will be created in Task 2). Wrap RootNavigator in the correct nesting order:
       ```
       ThemeProvider > AuthProvider > QueryClientProvider > ToastProvider > ModalProvider > RootNavigator
       ```
       Per RESEARCH.md Pattern 2 rationale: Auth wraps QueryClient (queries need auth), QueryClient wraps Toast/Modal (callbacks may trigger invalidation).
  </action>

    - Root layout wraps app with QueryClientProvider + ToastProvider + ModalProvider in correct order

    - ToastProvider: auto-dismiss after 4 seconds via setTimeout
    - ToastProvider: hideToast() dismisses immediately and clears timeout
    - ToastProvider: pressing dismiss on Alert calls hideToast
    - ToastProvider: renders toast as absolute-positioned Alert component overlaying content
    - ToastProvider: useToast() throws if used outside provider
    - ModalProvider: showModal({type, title, content?, onConfirm?, onCancel?}) shows modal
    - ModalProvider: hideModal() closes the modal
    - ModalProvider: confirmation type renders title + content + Cancel/Confirm buttons
    - ModalProvider: generic type renders title + content + Close button
    - ModalProvider: onConfirm callback fires when Confirm pressed, then modal closes
    - ModalProvider: onCancel callback fires when Cancel pressed, then modal closes
    - ModalProvider: useModal() throws if used outside provider
  </behavior>
  <action>
    1. Create `src/contexts/toast.tsx` following RESEARCH.md Pattern 5 and logbook reference:
       - ToastType = 'success' | 'info' | 'warning' | 'error'
       - Toast interface: { type: ToastType, title?: string, message: string }
       - ToastContextValue: { toast: Toast | null, showToast: (t: Toast) => void, hideToast: () => void }
       - Replace-not-append behavior: showToast clears existing timeout, sets new toast, starts 4s auto-dismiss
       - hideToast clears timeout, sets toast to null
       - Render toast overlay: position absolute, top of screen (top-4), full width, z-50, px-4
       - Use existing Alert component with visible={!!toast}, onDismiss={hideToast}
       - Platform-aware: on web position fixed at top-right (max-w-sm right-4), on mobile full-width top
       - Export ToastProvider and useToast hook

    2. Create `src/contexts/modal.tsx` following RESEARCH.md Pattern 6 and logbook reference:
       - ModalType = 'confirmation' | 'generic'
       - ModalConfig: { type: ModalType, title: string, content?: React.ReactNode, onConfirm?: () => void, onCancel?: () => void }
       - ModalContextValue: { showModal: (config: ModalConfig) => void, hideModal: () => void }
       - showModal sets config state, hideModal clears it
       - Renders using existing Modal (visible={!!config}) and ModalContent (title, children=content)
       - For 'confirmation' type: footer has Cancel (outlined) and Confirm (contained/danger) Buttons
       - For 'generic' type: footer has single Close (outlined) Button
       - Cancel calls onCancel then hideModal, Confirm calls onConfirm then hideModal
       - Export ModalProvider and useModal hook

    3. Extend `tests/__mocks__/supabase.ts`: Add `.from()` chain mock for data query operations:
       - `from(table)` returns object with `select()`, `insert()`, `update()`, `delete()` methods
       - Each method returns chainable object with `.eq()`, `.ilike()`, `.range()`, `.order()`, `.single()`, `.select()`
       - Final resolution returns `{ data: [], error: null, count: 0 }`
       - Keep existing auth mocks intact
